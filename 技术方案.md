# 大学生新生报到智能助手 - 技术方案

## 一、执行摘要

本方案针对大学生新生报到场景，设计并实现了一套完整的智能报到助手系统。系统采用Django + SQLite技术栈，实现了学生端和管理端两大核心功能模块，涵盖新生注册、报到任务管理、宿舍选择、校园地图导航和智能问答等核心功能。系统支持500人以下用户规模，采用前后端分离架构，API设计遵循RESTful规范，确保系统稳定、安全、易于维护。

***

## 二、系统架构设计

### 2.1 总体架构

#### 2.1.1 架构模式

采用**前后端分离架构**：

- **前端表现层**：HTML5 + CSS3 + JavaScript 响应式前端，支持PC和移动端访问
- **后端业务逻辑层**：Django Web框架处理业务逻辑、路由控制、数据处理
- **数据持久层**：SQLite数据库存储， Django ORM简化数据操作

**架构说明**：

1. **前端层**：
   - 学生端：信息管理、任务管理、到校登记
   - 管理员端：统计分析、学生管理、宿舍管理
   - 通过HTTP请求（GET/POST）与后端API交互

2. **API层**：
   - 为前端页面提供标准数据接口
   - 实现前后端分离，高效开发
   - 封装Django的业务逻辑

3. **Django后端**：
   - 认证模块：处理用户登录、注册、身份验证
   - 学生模块：管理学生信息、报到任务、到校登记
   - 宿舍模块：处理宿舍分配、管理
   - 问答模块：智能问答、FAQ管理

4. **ORM层**：
   - 用Python对象操作数据库
   - 自动生成SQL语句
   - 兼容多种数据库，方便后续扩展到MySQL

5. **数据库层**：
   - 学生表（Student）
   - 报到任务表（RegistrationTask）
   - 常见问题表（FAQ）
   - 宿舍表（Dormitory）
   - 学生宿舍分配表（StudentDormitory）

**技术优势**：
- Django内置多种模块功能，开发效率高
- ORM系统简化数据库操作，安全性好
- 前后端分离架构，易于维护和扩展
- 后续可无缝扩展到MySQL数据库

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              业务流程                                      │
└─────────────────────────────────────────────────────────────────────────────┘

1. 用户访问系统 → 进入 login-select.html (登录选择页面)
   │
   ↓
2. 选择身份 → 跳转到 login.html (学生端) 或 admin-login.html (管理员端)
   │
   ↓
3. 登录验证
   │
   ├─ 学生登录 → 调用 /api/student/register/ 进行身份验证
   │
   └─ 管理员登录 → 直接进入管理端 (简化验证流程)
   │
   ↓
4. 学生操作流程：
   │
   ├─ 查看任务 → 调用 /api/student/tasks/ 获取任务列表
   │
   ├─ 完成任务 → 调用 /api/task/update/ 更新任务状态
   │
   ├─ 编辑信息 → 调用 /api/student/update/ 修改个人信息
   │
   ├─ 到校登记 → 调用 /api/student/arrival/ 记录到校信息
   │
   └─ 选择宿舍 → 调用 /api/student/dorm-select/ 选择宿舍
   │
   ↓
5. 管理员操作流程：
   │
   ├─ 查看总览 → 调用 /api/admin/dashboard/ 获取统计数据
   │
   ├─ 查看学生详情 → 调用 /api/admin/student-detail/ 获取详细信息
   │
   ├─ 编辑学生 → 调用 /api/admin/edit-student/ 修改学生信息
   │
   ├─ 强制完成报到 → 调用 /api/admin/force-complete/ 强制完成
   │
   └─ 管理宿舍 → 调用 /api/admin/dorms/ 进行宿舍管理
   │
   ↓
6. 智能问答流程：
   │
   ├─ 用户输入问题 → 调用 /api/chat/ 发送问答请求
   │
   ├─ 关键词匹配 → 在FAQ数据库中查找匹配问题
   │
   └─ 返回回答 → 显示智能问答结果
   │
   ↓
7. 核心处理流程：
   │
   ├─ 前端发送HTTP请求 (GET/POST)
   │
   ├─ URL路由分发 → 匹配到对应的视图函数
   │
   ├─ 视图层处理 → 执行业务逻辑
   │
   ├─ 模型层操作 → 通过Django ORM操作数据库
   │
   └─ 返回响应 → JSON格式数据返回前端
   │
   ↓
8. 前端更新页面 → 显示操作结果 → 完成业务流程
```

**流程说明**：

1. **用户访问阶段**：用户访问系统入口 login-select.html，根据身份选择进入学生端或管理端

2. **身份验证阶段**：学生通过学号、姓名、身份证号进行验证；管理员通过简化验证直接进入

3. **业务操作阶段**：学生可以完成报到任务、修改信息、选择宿舍等操作；管理员可以查看统计数据、管理学生、分配宿舍等操作

4. **数据处理阶段**：前端通过API与后端交互；Django接收请求后进行业务处理；通过ORM操作SQLite数据库

5. **响应返回阶段**：后端返回JSON格式响应；前端更新页面显示操作结果

## 三、技术实现方案

### 3.1 数据库设计

#### 3.1.1 数据模型关系

```
Student (学生表)
    └── 1:N → RegistrationTask (报到任务表)

Student (学生表)
    └── 1:1 → StudentDormitory (宿舍分配表)
              └── N:1 → Dormitory (宿舍表)

FAQ (常见问题表) - 独立表，无外键关联
```

#### 3.1.2 核心表结构

##### 学生表（Student）
| 字段名       | 类型          | 约束        | 说明         |
| ---------- | ----------- | --------- | ---------- |
| student_id | CharField   | 主键       | 学号        |
| name       | CharField   | 非空       | 姓名        |
| id_card    | CharField   | 非空、唯一    | 身份证号      |
| major      | CharField   | 非空       | 录取专业      |
| college    | CharField   | 默认值      | 学院归属      |
| phone      | CharField   | 非空       | 手机号       |
| status     | CharField   | 默认值      | 报到状态      |
| created_at | DateTimeField | 自动创建   | 创建时间      |
| updated_at | DateTimeField | 自动更新   | 更新时间      |

**状态选项**：UNREGISTERED（未报到）、IN_PROGRESS（进行中）、COMPLETED（已完成）

##### 报到任务表（RegistrationTask）
| 字段名         | 类型           | 约束        | 说明        |
| ---------- | -------------- | --------- | --------- |
| id         | AutoField      | 主键       | 任务ID      |
| student    | ForeignKey     | 外键、非空   | 关联学生     |
| task_name  | CharField      | 非空       | 任务名称     |
| status     | CharField      | 默认值      | 任务状态     |
| completed_time | DateTimeField | 可为空     | 完成时间     |
| created_at | DateTimeField | 自动创建    | 创建时间     |
| updated_at | DateTimeField | 自动更新    | 更新时间     |

**状态选项**：PENDING（未完成）、COMPLETED（已完成）

##### 常见问题表（FAQ）
| 字段名       | 类型          | 约束     | 说明      |
| ---------- | ----------- | ------ | ------- |
| id         | AutoField   | 主键    | 问题ID    |
| question   | TextField   | 非空    | 问题内容   |
| answer     | TextField   | 非空    | 回答内容   |
| category   | CharField   | 非空    | 分类      |
| created_at | DateTimeField | 自动创建 | 创建时间   |
| updated_at | DateTimeField | 自动更新 | 更新时间   |

**分类选项**：宿舍生活、饮食、校园网、快递、学习、学籍管理、图书馆、体育课、军训、财务、报到流程

##### 宿舍表（Dormitory）
| 字段名           | 类型          | 约束     | 说明    |
| -------------- | ----------- | ------ | ----- |
| id             | AutoField   | 主键    | 宿舍ID  |
| building       | CharField   | 非空    | 宿舍楼   |
| room           | CharField   | 非空    | 房间号   |
| total_beds     | IntegerField | 默认值4  | 总床位   |
| available_beds | IntegerField | 默认值4  | 剩余床位  |
| is_occupied    | BooleanField | 默认False | 是否已满 |
| created_at     | DateTimeField | 自动创建 | 创建时间 |
| updated_at     | DateTimeField | 自动更新 | 更新时间 |

##### 学生宿舍分配表（StudentDormitory）
| 字段名       | 类型           | 约束      | 说明     |
| ---------- | -------------- | ------- | ------ |
| id         | AutoField      | 主键     | 分配ID   |
| student    | OneToOneField  | 外键、唯一  | 关联学生  |
| dormitory  | ForeignKey     | 外键、可空  | 关联宿舍  |
| assigned_at | DateTimeField | 自动创建   | 分配时间  |

### 3.2 安全设计

#### 3.2.1 认证与授权

- **身份验证**：学生通过学号、姓名、身份证号进行身份确认
- **Session管理**：Django内置Session机制管理用户状态
- **权限控制**：前后端分离的权限验证
  - 学生角色：访问学生端功能
  - 管理员角色：访问管理端功能

#### 3.2.2 数据安全

- **SQL注入防护**：使用Django ORM参数化查询
- **XSS防护**：模板系统自动转义
- **CSRF防护**：对API接口添加@csrf_exempt装饰器
- **数据验证**：所有用户输入进行严格验证
- **错误处理**：统一的错误响应格式，不暴露敏感信息

#### 3.2.3 网络安全

- **HTTP协议**：开发环境使用HTTP
- **Session安全**：敏感信息不直接暴露
- **输入验证**：前后端双重验证机制

### 3.3 性能优化

#### 3.3.1 数据库优化

- **索引设计**：在常用查询字段建立索引
  - Student.student_id（主键索引）
  - Student.id_card（唯一索引）
  - RegistrationTask.student（外键索引）
  - Dormitory.building（查询索引）
- **查询优化**：
  - 使用Django ORM的filter和get方法
  - 使用related_name优化关联查询
  - 合理使用select_related减少查询次数

#### 3.3.2 前端优化

- **静态资源**：本地存储，无需CDN
- **异步加载**：fetch API异步请求
- **缓存策略**：localStorage缓存用户信息
- **代码压缩**：精简JavaScript代码

#### 3.3.3 性能指标

- 页面加载时间：< 3秒
- 数据库查询响应：< 1秒
- 并发支持：50-100用户同时在线

### 3.4 可用性设计

#### 3.4.1 响应式设计

- **移动端适配**：CSS Flexbox和Grid布局
- **触摸友好**：按钮和链接尺寸优化
- **浏览器兼容**：支持Chrome、Firefox、Edge、Safari

#### 3.4.2 用户体验

- **操作反馈**：alert和prompt提示操作结果
- **表单验证**：前端和后端双重验证
- **错误处理**：友好的错误提示
- **进度展示**：报到进度可视化展示

### 3.5 智能问答方案

#### 3.5.1 技术实现

- **问答匹配**：基于关键词匹配规则
- **FAQ查询**：从数据库查询常见问题
- **上下文处理**：维护对话上下文

#### 3.5.2 功能特性
